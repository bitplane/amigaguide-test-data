@NODE "--background--" "bbsread.library/--background--"

    @{b}NOTES@{ub}
	Since this library uses functions in the dos.library, only processes
	should open and use this library.

	All strings passed to functions in this library should use the 
	standard Amiga character set (ISO). Use @{"CharsetConvert()" LINK "CharsetConvert" 0} to convert 
	the charset of strings which is not in ISO.

	The library uses two envirorment variables: 
	o THOR/BBSDataPath - Path to where to store the database and 
	    configuration files.
	o THOR/THORPath - Path to where the Thor system is installed. This
	    path *must* end with a ':' or a '/'.

	Progress Hooks:
	Several functions support call back progress hooks. The hook is 
	called with a @{"BRProgress" LINK "libraries/bbsread.h/MAIN" 831} structure as the object parameter. The 
	@{"BRProgress" LINK "libraries/bbsread.h/MAIN" 831} structure is describred in <libraries/bbsread.h>. The 
	tags for progress hooks are global:
	
	BR_ProgressHook - Callback hook for progress report is pointed by
	    (struct @{"Hook" LINK "libraries/xpk.h/MAIN" 43} *) ti_Data. Returning non-zero from the callback hook
	    will stopp the operation and in some cases make the function
	    return failure.

	BR_ProgressUpdates - The maximum number of calls to the callback hook
	    for one pass is in (@{"ULONG" LINK "exec/types.h/MAIN" 36}) ti_Data. The default is 100.

	BR_ProgressReturn - Pointer to a @{"ULONG" LINK "exec/types.h/MAIN" 36} to store the return value from
	    the progress hook is in (@{"ULONG" LINK "exec/types.h/MAIN" 36} *) ti_Data.

	Common configuration tags: 
	(Not supported by all configuration functions yet.)

	BRCFG_Use - Don't store the changes done to the configuration if 
	    (@{"BOOL" LINK "exec/types.h/MAIN" 68}) ti_Data is TRUE.

	BRCFG_LastSaved - Will retrive the configuration last saved before 
	    applying any changes.

   @{b}SYNOPSIS@{ub}

   @{b}FUNCTION@{ub}
	This function add files to an archive. The type of archiver to use is
	determined with the arcitem. 
	
	The ability to return failure when the archiver fails depends on how 
	the archivers behave.

	The tags not understood are forwarded to dos.library/SystemTagList(). 
	Look at dos.library/SystemTagList() for futher information on tags.

   @{b}INPUTS@{ub}
	arcitem - Pointer to struct @{"ArcConfigItem" LINK "libraries/bbsread.h/MAIN" 40} to tell which archiver to 
	    use.

	archive - Pointer to path and name of archive to add to.
	tagitems - Pointer to @{"TagItem" LINK "libraries/xpk.h/MAIN" 34} array. See <@{"dos/dostags.h" LINK "dos/dostags.h/MAIN" 0}>.  Both 
	    dos.library/SystemTagList() tags and dos.library/CreateNewProc()
	    tags may be passed.

	Here are the TagItem.ti_Tag values that are defined for
	Archive().

	AR_AddFile - File to add to archive is in (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) ti_Data. At least 
	    one of this tag must be passed.
	
	AR_SourceDir - Path to source directory is in (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) it_Data. 
	    Default is to use current directory as source directory.

   @{b}RESULT@{ub}
	error	- 0 for success, result from archiver, or -1.  Note that on
	    error, the caller is responsible for any filehandles or other
	    things passed in via tags.

@NODE "BufBROpen" "bbsread.library/BufBROpen()"

   @{b}NAME@{ub}	
	BufBROpen -- Open a BBSRead buffered file for input or output.

   @{b}SYNOPSIS@{ub}
	fileid = BufBROpen( name, accessMode )
	D0                   A0       D0

	@{"APTR" LINK "exec/types.h/MAIN" 33} BufBROpen( @{"STRPTR" LINK "exec/types.h/MAIN" 51}, @{"LONG" LINK "exec/types.h/MAIN" 35} );

   @{b}FUNCTION@{ub}
	The named file is prepared to be used for input or output through the 
	BBSRead buffering system. All files are opened in shared mode. Uses 
	the same mode constants as dos.library/Open().

	BBSRead buffered files should *not* be used when you want to read and
	write large blocks.

	BBSRead buffered files should *not* be written to by processes not 
	using @{"BufBRWrite()" LINK "BufBRWrite" 0}.

   @{b}INPUTS@{ub}
	name - Name of file to open.
	accessMode - Mode to open file in.
		

   @{b}RESULT@{ub}
	fileid - The fileid for the opened file. Returns a NULL pointer if 
	    the open failed, and a secondary error code will be available in 
	    IoErr().

   @{b}EXAMPLE@{ub}

   @{b}NOTES@{ub}
	The fileid returned is *not* compatible with the file handler 
	returned by dos.library/Open().

   @{b}SEE@{ub} @{b}ALSO@{ub}

   @{b}FUNCTION@{ub}
	Changes the setup for a BBS, or adds a new BBS to the database. 

   @{b}INPUTS@{ub}
	bbs - Pointer to the @{"BBSListItem" LINK "libraries/bbsread.h/MAIN" 413} for the bbs to change configuration 
	    for. If this pointer is a NULL pointer, a new bbs will be 
	    created. The contents of bbs->bl_Data will be updated and string 
	    pointers may change.
	tagitems - Pointer to @{"TagItem" LINK "libraries/xpk.h/MAIN" 34} array.

	Here are the TagItem.ti_Tag values that are defined for
	ConfigBBS().

	BC_DeleteBBS - Markes the BBS pointed by the bbs parameter as deleted
	    if (@{"BOOL" LINK "exec/types.h/MAIN" 68}) ti_Data is TRUE. Your @{"BBSListItem" LINK "libraries/bbsread.h/MAIN" 413} passed as the bbs 
	    parameter will be removed from your bbslist. The pointer returned
	    is the old bbs pointer, but it will not point to a @{"BBSListItem" LINK "libraries/bbsread.h/MAIN" 413} any
	    more.

	    ti_Data. This is the pointer returned from @{"GetBBSList()" LINK "GetBBSList" 0}. When
	    adding a new BBS to the database this tag *must* be supplied. 
	    It *must* also be supplied when using the BC_Top, BC_Bottom,
	    BC_Up, BC_Down, BC_SortBBSList and BC_NewBBSOrder tags. 

	BC_BBSName - The new name of the BBS is pointed by (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) ti_Data.
	BC_GrabName - The new name of the grabfile (without extension) is 
	    pointed by (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) ti_Data. Wildcards are supported. If 
	    wildcards are used, must the name match the full file name.

	BC_BBSType - The new type of the BBS is pointed by (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) ti_Data.
	    If no BBSType with this name is defined, the funktion will fail.

	BC_UserName - The new name the user is registered as is pointed by 
	    (@{"STRPTR" LINK "exec/types.h/MAIN" 51}) ti_Data.

	BC_ScriptFlags - Flags for the script is in (@{"LONGBITS" LINK "exec/types.h/MAIN" 37}) ti_Data. Check
	    the BBS type for available flags. Non-available flags passed will
	    be ignored.

	BC_Signature - Signature to use on this BBS is pointed by (@{"STRPTR" LINK "exec/types.h/MAIN" 51})
	    ti_Data. This signature should be used in conferences whith no
	    signature defined. A NULL-pointer equals no BBS signature, and the
	    global signature should be used on this BBS. If the signature is
	    in a file, this tag should contain the complete path to the
	    signature file, and the BDF_FILE_SIGNATURE flag must be set.

	BC_Top - Move the BBS to the top of the list if (@{"BOOL" LINK "exec/types.h/MAIN" 68}) ti_Data is 
	    TRUE. Both your local and the global list will be updated. Be 
	    sure to also pass the BC_BBSList tag.

	BC_Bottom - Move the BBS to the bottom of the list if (@{"BOOL" LINK "exec/types.h/MAIN" 68}) ti_Data
	    is TRUE. Both your local and the global list will be updated. Be
	    sure to also pass the BC_BBSList tag.

	BC_Up - Move the BBS one position upwards on the list if (@{"BOOL" LINK "exec/types.h/MAIN" 68}) 
	    ti_Data is TRUE. Both your local and the global list will be
	    updated. Be sure to also pass the BC_BBSList tag.

	BC_Down - Move the BBS one position downwards on the list if (@{"BOOL" LINK "exec/types.h/MAIN" 68})
	    ti_Data is TRUE. Both your local and the global list will be
	    updated. Be sure to also pass the BC_BBSList tag.

	BC_SortBBSList - Sort the BBS-list alphabetically if (@{"BOOL" LINK "exec/types.h/MAIN" 68}) ti_Data is
	    TRUE. Both your local and the global list will be sorted. Be sure
	    to also pass the BC_BBSList tag.
